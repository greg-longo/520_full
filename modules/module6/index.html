What is Version Control?

A version control system records every change you make to your files over time, so you can go back to any earlier state of your project. Think of it as an unlimited undo history - not just for one file, but for your entire project directory.

For data scientists, this matters because your work is rarely linear. You'll try an approach, hit a dead end, and need to backtrack. You'll share a notebook with a teammate who changes it in ways that break your analysis. You'll want to know exactly what changed between the version that worked and the version that didn't. Version control handles all of this.

Git is the most widely used version control system in the world. GitHub is a hosting platform built on top of Git that adds collaboration features like pull requests, code review, and issue tracking.

Using the Terminal

Git is a command-line tool - you interact with it by typing commands into a text window called a terminal, not by clicking through a graphical interface. Most developers prefer this because it's faster, scriptable, and it's exactly how Git is used in professional environments.

The terminal runs a program called a shell that interprets your commands. The shell you'll use depends on your operating system:

Mac - Terminal (zsh)

macOS ships with the Terminal app running zsh. You can also use iTerm2 as a more feature-rich alternative. All commands in this course work on Mac Terminal.

Windows - Git Bash

Windows uses PowerShell or CMD by default, but Git Bash (installed with Git for Windows) gives you a Bash shell that uses the same commands as Mac and Linux.

VS Code - Integrated Terminal

VS Code has a built-in terminal (View > Terminal or Ctrl+`) that opens in your project folder. This is the most convenient option when you're already working in VS Code.

Opening Your Terminal

On a Mac

Press Cmd + Space to open Spotlight, type Terminal, and press Enter. Or find it in Applications > Utilities > Terminal. Your prompt will look like: intern@macbook ~ %

On Windows

Download and install Git for Windows from git-scm.com. Once installed, right-click any folder and choose "Open Git Bash here", or search for Git Bash in the Start menu.

Using GitHub Codespaces

If you're working in GitHub Codespaces (a cloud-based VS Code environment), your terminal is already set up. Open the integrated terminal with Ctrl+` and you're ready to run Git commands - no local installation needed.

The Three States

Before diving into the states, it helps to know how Git stores your work. Unlike tools that track a list of changes ("line 14 changed from X to Y"), Git saves a complete snapshot of every file in your project each time you commit - like taking a photograph of the entire folder at that moment. This is why reverting to any point in history is instant: Git just loads the snapshot from that commit rather than replaying a chain of diffs.

This snapshot model is also why git add and git commit are two separate steps. You're first choosing what goes into the snapshot (staging), then pressing the shutter (committing). Every file in a Git project lives in one of three states that reflect where it is in that process:

Working Directory

Your actual files on disk. When you edit a notebook or Python script, the changes live here until you stage them.

Staging Area

A holding zone for changes you're about to commit. You choose which changes to include - not every edit has to go into the same commit.

Repository

The committed history. Once changes are committed, Git permanently records a snapshot. This is your undo history.

Remote (GitHub)

A copy of your repository hosted on GitHub. Push to share your commits with others; pull to receive theirs.

The Core Workflow

Almost every Git session follows the same pattern. Once you have a repository set up, the day-to-day cycle looks like this:

# 1. Pull in any changes your teammates made
git pull origin main

# 2. Make changes to your files (edit, add, delete)

# 3. Stage the changes you want to include
git add analysis.py          # stage one file
git add .                    # stage everything

# 4. Commit with a descriptive message
git commit -m "Add batting average normalization step"

# 5. Push your commits to GitHub
git push origin main
Good commit messages

Write commit messages as if you're finishing the sentence "This commit will..." - so "Add batting average normalization step" rather than "stuff" or "fixed it". Your future self will thank you when you're scrolling through history three months later.

Setting Up a Repository

There are three common ways to start a Git project depending on your situation: initialize a brand new one locally, clone an existing one from GitHub, or fork someone else's repo and clone your copy.

Option A: Start locally (local-first)

Use this when you're starting a brand new project from scratch on your own machine. You create the folder, initialize Git, make your first commit, then connect it to GitHub.

# 1. Create a folder and navigate into it
mkdir my-project
cd my-project

# 2. Initialize Git
git init

# 3. Add files and make your first commit
git add .
git commit -m "Initial commit"

# 4. Connect to a GitHub repo you already created on github.com
git remote add origin https://github.com/your-username/your-repo.git
git push -u origin main

Option B: Start from GitHub (remote-first)

Use this when the repo already exists on GitHub - either one you created there, or one your instructor set up. git clone downloads the entire repo, history and all, and wires up the remote automatically.

# Clone creates the folder for you
git clone https://github.com/your-username/your-repo.git
cd your-repo

# You're ready to work - remote is already configured
git remote -v   # verify the remote is set

Option C: Fork then clone (course workflow)

Forking creates your own independent copy of someone else's repository under your GitHub account. You can push changes to your fork freely without affecting the original. This is the standard workflow for course assignments and open-source contributions.

The steps are always the same:

Fork + Clone Workflow

1. On GitHub, open the repo you want to work with and click the Fork button (top right). GitHub creates a copy at github.com/your-username/repo-name.

2. Clone your fork (not the original) to your machine:

git clone https://github.com/your-username/repo-name.git
cd repo-name

Any commits you push go to your fork. If you want to propose changes back to the original, you open a pull request from your fork - the same PR workflow covered in the Pull Requests section below.

Clone your fork, not the original

A common mistake: cloning the original repo instead of your fork. If you do that, you won't have push access. Always copy the URL from your forked version of the repo on GitHub.

Branching

A branch is an independent copy of your project history that you can work on without touching the main codebase. When you're building a new feature, fixing a bug, or running an experiment, you do that work on a branch. If it goes well, you merge it back. If it doesn't, you discard the branch and nothing is lost. 

# See all branches (the * marks your current branch)
git branch

# Create a new branch and switch to it
git checkout -b feature/add-salary-viz

# Do your work, commit as normal, then push the branch
git push origin feature/add-salary-viz

# Switch back to main
git checkout main

The convention is to keep main always in a working, deployable state. All active development happens on branches. This is how professional teams prevent broken code from reaching the shared codebase.

Merging and Conflicts

When your branch is ready, you merge it back into main. Git will try to combine the changes automatically. If two branches modified the same lines of the same file, Git cannot decide which version is correct - that's a merge conflict. 

# Merge a branch into main
git checkout main
git merge feature/add-salary-viz

# If a conflict occurs, Git marks the file:
# <<<<<<< HEAD
# your version of the code
# =======
# the incoming version
# >>>>>>> feature/add-salary-viz

# Edit the file to resolve, then:
git add analysis.py
git commit -m "Merge salary viz feature, resolve conflict"
Conflicts aren't failures

Merge conflicts are a normal part of collaborative work - not a sign something went wrong. Git is simply telling you it needs a human to decide which version of the code is correct. Read both sides carefully before choosing.

7 / 9

Pull Requests

On GitHub, you don't usually merge branches directly from the command line when working on a team. Instead, you open a Pull Request (PR) - a proposal to merge your branch into main, with a built-in discussion thread for code review.

The pull request workflow looks like this:

Push your branch to GitHub, then open a PR through the GitHub interface. A teammate reviews your changes, leaves comments, and either approves or requests changes. Once approved, the branch is merged and you delete the branch to keep the repo clean.

Best practices recap

Always git pull before you start new work. Write clear commit messages. Use branches for every new feature or fix. Review your own PR before asking someone else to. Delete merged branches.
