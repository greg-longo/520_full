What is Data Science?

Data science is the practice of extracting knowledge and insight from data through a combination of statistics, programming, and domain expertise. The oft-cited "Venn diagram" definition says data science sits at the intersection of three areas: math and statistics, computer science and programming, and domain knowledge. You need at least two of the three to be effective, and all three to be exceptional.

A more practical definition: data scientists build systems that turn raw data into decisions. Those decisions might be automated (a recommendation engine, a fraud detection model) or human-assisted (a dashboard that helps an executive understand churn). Either way, the job is to move from messy, ambiguous data toward a clear, defensible answer.

The DS Workflow
Most data science work follows a repeating cycle: Ask a specific, answerable question → Acquire data that could answer it → Clean and transform it → Explore to understand patterns → Model or analyze → Communicate findings. You will cycle through these steps - often multiple times - on every real project.

Data science vs. data engineering vs. ML engineering

The field has fragmented into specializations. Data scientists build models and generate insight. Data engineers build the pipelines that move and store data at scale. ML engineers take a data scientist's prototype model and deploy it into production systems. In a small company, one person often does all three; at large companies they are distinct teams.

Data Science vs. Data Analytics

These two terms are often used interchangeably, but they have a meaningful distinction. Understanding the difference helps you know what kind of work you are being asked to do.

DimensionData AnalyticsData Science
Primary question "What happened? Why did it happen?" "What will happen? What should we do?"
Time orientation Backward-looking (descriptive) Forward-looking (predictive / prescriptive)
Core methods Aggregation, segmentation, dashboards, SQL Machine learning, statistical modeling, experimentation
Output Reports, charts, KPI dashboards Trained models, automated decisions, recommendations
Skills emphasis SQL, Excel/Tableau, business communication Python/R, ML libraries, statistics, software engineering
Example at a baseball team Which players improved their on-base percentage this season? Which minor-league players are most likely to succeed at the MLB level?

In practice, analytics and data science are complementary. Analysts keep the lights on - they make sure the business understands what is happening right now. Data scientists push the frontier - they build systems that automate and scale decisions. Most teams need both.

Careers in Data

The data field offers a wide range of roles. Here are the most common entry points and mid-career positions, mapped to the skills this course builds toward.

Data Analyst

Turns business questions into SQL queries and visualizations. Communicates trends to non-technical stakeholders. Often the best entry-level path for career changers. Median salary ~$70-90k.

Data Scientist

Builds predictive models, designs experiments (A/B tests), and develops data products. Requires stronger Python and statistics skills. Median salary ~$110-140k.

Data Engineer

Designs and maintains the pipelines that collect, store, and move data. Heavy focus on SQL, cloud platforms (AWS/GCP/Azure), and distributed systems. Median salary ~$120-150k.

ML Engineer

Takes models from prototype to production. Focuses on model serving, monitoring, and MLOps infrastructure. Requires the most software engineering depth. Median salary ~$140-180k.

Business Intelligence (BI) Developer

Builds and maintains dashboards and reporting infrastructure (Tableau, Power BI, Looker). Bridges the gap between raw data and business decision-makers.

Research Scientist

Conducts original research, often in academia or large tech R&D labs. Requires deep statistics and often a PhD. Publishes findings and advances the state of the art.

Which role is right for me?
If you enjoy storytelling and working with business stakeholders, analytics is a natural fit. If you enjoy coding and building systems, data engineering or ML engineering may suit you better. If you enjoy math and research, data science or research science is the direction to pursue. Most practitioners start in analytics and specialize over time.

Types of Companies Hiring Data Talent

The type of company you work for shapes what data science actually looks like day-to-day. A startup data scientist does very different work than one at a large tech company or a hospital system.

Company TypeData MaturityWhat DS Work Looks Like
Early-stage startup Low - often no data infra yet Generalist work: build dashboards, set up pipelines, and run quick analyses. High impact, high ambiguity.
Growth-stage tech company Medium - some tooling in place Mix of analytics and modeling. Often involved in experimentation (A/B tests) and building recommendation or personalization features.
Large tech company (FAANG-adjacent) High - mature ML platform Specialized, deep work. May own one model or one metric for a year. High scale, high rigor, slower iteration.
Consulting / agency Varies by client Breadth over depth. Work across many industries and problem types. Strong communication skills essential.
Healthcare / finance / government Varies, often constrained High regulatory requirements. Data is messy and access is restricted. Explainability and compliance matter more than raw model performance.
Sports analytics Growing rapidly Player evaluation, in-game decision support, ticket pricing, fan engagement. Combines domain expertise with statistical modeling.

The Data Science Tool Stack

No single tool does everything. A working data scientist typically uses a layered stack - each layer handles a different part of the workflow. Here are the tools you will encounter most often, and the ones this course prepares you to use.

Python

Primary language for data science. Covers everything from cleaning to modeling to deployment.

Jupyter Notebooks

Interactive coding environment. Mix code, output, and narrative in one document.

pandas

Tabular data manipulation. DataFrames, filtering, aggregation, and cleaning.

NumPy

Numerical computing. Fast array operations that power every other library.

Matplotlib / Seaborn

Visualization. Static charts, distribution plots, heatmaps.

scikit-learn

Machine learning. Classification, regression, clustering, model evaluation.

SQL

The universal language for querying databases. Essential for any data role.

Git / GitHub

Version control. Track code changes, collaborate, and deploy projects.

Tableau / Power BI

Business intelligence dashboards. Drag-and-drop visualization for non-coders.

This course focuses on the Python layer - pandas, NumPy, Matplotlib, and Git. These are the foundational tools that make the rest of the stack accessible.

Why Python?

Python has become the dominant language of data science. It was not designed specifically for data science - it is a general-purpose language - but a combination of factors made it the community standard.

The practical reasons

Readable syntax. Python reads almost like English pseudocode. A data scientist can spend their cognitive budget on the problem rather than the syntax. Compare writing a loop in Python vs. Java and the difference is immediately apparent.

The ecosystem. NumPy, pandas, Matplotlib, scikit-learn, TensorFlow, PyTorch - all Python-first. The open-source community has built an unmatched set of libraries for every step of the data pipeline. This creates a network effect: people use Python because the libraries are there; the libraries improve because people use Python.

Glue language. Python can call code written in C, Fortran, R, and Java. The heavy numerical computation in NumPy is actually running C code under the hood. You get ease of use at the surface and raw speed where it counts.

Community and employers. The majority of job postings for data science roles list Python as a required or preferred skill. Choosing Python means choosing the largest community, the most Stack Overflow answers, and the most tutorials.

Python vs. R

R is Python's main competitor in academic and statistical contexts. R was designed by statisticians for statistical computing and is excellent for certain tasks - particularly advanced statistical modeling and publication-quality plots. But Python has largely displaced R in industry because of its broader applicability. If you go into academic research or biostatistics, you may need R. For most industry data science roles, Python is the right starting point.

Critical Thinking for Data Scientists

Technical skills are necessary but not sufficient. A data scientist who cannot think critically about a problem will build the wrong model correctly - which is worse than building nothing at all. These mental habits separate effective practitioners from technically competent ones.

Ask the right question first

Most projects fail not because of poor modeling but because the question was never precisely stated. "How do we improve sales?" is not an answerable data science question. "Which customer segments have the highest 90-day churn rate, and what behaviors predict churn 30 days in advance?" is. Before touching data, you should be able to state: what decision will this analysis inform, and what would a good answer look like?

Correlation is not causation

This is the most-repeated phrase in statistics for a reason. Finding that two variables move together does not tell you why. Ice cream sales and drowning rates are correlated - because both increase in summer. Acting on that correlation would be absurd. Whenever you find a relationship in data, ask: what third variable could explain this? Could the causality run the other direction? Is this a coincidence in a small sample?

Know your data's provenance

Before analyzing data, understand how it was collected. Surveys have response bias. Clickstream data reflects only users who stayed on the page. Hospital admission data does not represent the general population. Every dataset has selection effects baked in. Ignoring them leads to confident, wrong conclusions.

Uncertainty is not failure

A good data scientist quantifies uncertainty and communicates it clearly. Saying "we are 80% confident that treatment B outperforms treatment A by 5-15%" is more honest and more useful than saying "B is better than A." Decision-makers need to know the confidence interval, not just the point estimate.

A 6-Step Problem-Solving Framework

Before writing a single line of Python, effective data scientists work through the problem systematically. This six-step framework builds the habit of thinking before coding - which saves far more time than it costs.

1. Understand

Comprehend requirements and constraints. Identify the inputs, outputs, and any specific rules or conditions that must be met.

2. Plan

Consider the logical flow. Make a general outline of the major steps. Think about which data structures or algorithms might be suitable.

3. Break It Down

Divide the plan into smaller, more manageable sub-problems. Each sub-problem should be solvable independently.

4. Pseudocode

Write a human-friendly description of what the program should do - without worrying about Python syntax yet.

5. Code Each Part

Implement each sub-problem in Python. You don't always have to start at the beginning - tackle the parts you understand first.

6. Test & Debug

Check that your solution works on expected inputs and edge cases. Learn to read error messages as instructions, not failures.

Pseudocode: Thinking Before Coding

Pseudocode is a structured plain-language description of an algorithm - formal enough to be unambiguous, but free of any specific programming language syntax. It forces you to think through the logic of a solution before you are distracted by syntax errors and edge cases.

Good data scientists write pseudocode before they open a Jupyter notebook. It is faster to iterate on an idea in plain language than to debug Python that implements the wrong algorithm.

Example: finding the best-performing player

Problem: Given a roster of players with batting averages, identify the player with the highest average who has played at least 20 games.

// Pseudocode - no Python syntax yet INPUT: roster (list of player records, each with name, avg, games_played) OUTPUT: name of the qualifying player with the highest batting average SET best_player = null SET best_avg = 0 FOR EACH player IN roster: IF player.games_played >= 20: IF player.avg > best_avg: SET best_player = player.name SET best_avg = player.avg RETURN best_player

Now translating to Python is mechanical - the thinking is already done:

def best_qualifying_player(roster, min_games=20):
    best_player = None
    best_avg = 0

    for player in roster:
        if player['games_played'] >= min_games:
            if player['avg'] > best_avg:
                best_player = player['name']
                best_avg = player['avg']

    return best_player


roster = [
    {'name': 'Rivera', 'avg': .312, 'games_played': 28},
    {'name': 'Chen',   'avg': .287, 'games_played': 15},  # excluded
    {'name': 'Walker', 'avg': .301, 'games_played': 30},
]

print(best_qualifying_player(roster))  # 'Rivera'
Pseudocode vs. comments
Pseudocode is written before the code exists. Comments are written inside existing code to explain it. Both are valuable, but pseudocode is a planning tool - it helps you discover edge cases and design flaws before you are invested in a half-written implementation. Get in the habit of sketching logic in plain language first.
